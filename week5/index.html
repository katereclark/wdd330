Week 5 Notes

Questions

  • What are some other good TDD JS libraries?

Notes

  • System error ― there is a problem with the system or external devices with which the program is interacting.
  • Programmer error ― the program contains incorrect syntax or faulty logic; it could even be as simple as a typo.
  • User error ― the user has entered data incorrectly, which the program is unable to handle.
  • Ensure that code fails loudly in development so errors can be found quickly
  • Make code fail gracefully but not completely silently, and so as to not affect user experience
  • Strict mode - produces more exceptions and warnings and prohibits the use of some deprecated features - throws errors if sloppy code is used
  • 'use strict'
  • self-invoking function - (function() { 'use strict';}());
  • Linting tools (lint test) - highlight any sloppy programming practices or syntax errors, and will complain if certain style conventions are not followed - enforce style guide
  • The recommended way to determine browser support for a feature is to use feature detection. This is done using an if statement to check whether an object or method exists before trying to actually call the method.
  • Modernizr is a library that offers an easy way to implement feature detection and Can I Use? is another useful resource for checking which features are currently supported in different browsers.
  • In many cases, the point at which an error occurs is not always where it originated
  • The most basic form of debugging is to use the alert() method to show a dialog at certain points in the code - alert() stops a program from running until OK is clicked
  • console.trace() method - will log an interactive stack trace in the console.
  • debugger keyword - will create a breakpoint in your code that will pause the execution of the code and allow you to see where the program is currently up to
  • An error object can be created by the host environment when an exception occurs, or it can be created in the code using a constructor function
  • EvalError, RangeError, ReferenceError, SyntaxError, TypeError, URIError, InternalError
  • These error objects can also be used as constructors to create custom error objects: const error = new TypeError('You need to use numbers in this function');
  • throw statement - throw new Error('Something has gone badly wrong!');
  • try, catch, anf finally
  • Testdriven development (TDD) - write tests that initially fail, write code to pass the tests, refactor the code, test refactored code, write more test for new features (red-green-refactor)
  • testing frameworks - Jest ( jest -c {} ), run files that end in test.js within that folder
  • test('non-numerical data returns not prime', () => { expect(isPrime('two')).toBe(false); });
